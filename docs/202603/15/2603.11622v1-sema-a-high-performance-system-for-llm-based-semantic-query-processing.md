## 摘要
大语言模型（LLM）与数据分析的集成，为海量结构化和非结构化数据的推理释放了强大的能力。然而，现有系统通常依赖于 DataFrame 原语（缺乏现代数据库管理系统的高效执行基础设施）或 SQL 用户自定义函数（UDF，将语义逻辑与查询优化器隔离，并增加了用户的实现复杂性）。由于 LLM 调用的高昂成本和非确定性，由 LLM 驱动的语义算子也带来了新的挑战，传统的优化规则和代价模型无法适用于这些算子的优化。为了弥补这些差距，我们提出了 Sema，这是一个构建在 DuckDB 之上的高性能语义查询引擎，它将 LLM 驱动的语义算子视为“一等公民”。Sema 引入了 SemaSQL，这是一种声明式方言，允许用户将自然语言表达式无缝注入标准 SQL 子句中，从而实现端到端的优化和执行。在逻辑层面，Sema 的优化器会压缩自然语言表达式，并从语义算子中推导出关系约束。在运行时，Sema 采用自适应查询执行（AQE）来动态重排算子、融合语义操作并应用提示词批处理。该方法旨在准确性约束下，寻求平衡 Token 消耗和延迟的帕累托最优执行路径。我们在涵盖分类、摘要和提取任务的 20 个语义查询上对 Sema 进行了评估。实验结果表明，与三个基准系统相比，Sema 实现了 2-10 倍的加速，同时保持了具有竞争力的结果质量。

## Abstract
The integration of Large Language Models (LLMs) into data analytics has unlocked powerful capabilities for reasoning over bulk structured and unstructured data. However, existing systems typically rely on either DataFrame primitives, which lack the efficient execution infrastructure of modern DBMSs, or SQL User-Defined Functions (UDFs), which isolate semantic logic from the query optimizer and burden users with implementation complexities. The LLM-powered semantic operators also bring new challenges due to the high cost and non-deterministic nature of LLM invocation, where conventional optimization rules and cost models are inapplicable for their optimization.   To bridge these gaps, we present Sema, a high-performance semantic query engine built on DuckDB that treats LLM-powered semantic operators as first-class citizens. Sema introduces SemaSQL, a declarative dialect that allows users seamlessly inject natural language expressions into standard SQL clauses, enabling end-to-end optimization and execution. At the logical level, the optimizer of Sema compresses natural language expressions and deduces relational constraints from semantic operators. At runtime, Sema employs Adaptive Query Execution (AQE) to dynamically reorder operators, fuse semantic operations, and apply prompt batching. This approach seeks a Pareto-optimal execution path balancing token consumption and latency under accuracy constraints. We evaluate Sema on 20 semantic queries across classification, summarization, and extraction tasks. Experimental results demonstrate that Sema achieves $2-10 \times$ speedup against three baseline systems while achieving competitive result quality.

### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）在数据分析中的应用日益广泛，如何高效处理涉及结构化和非结构化数据的“语义查询”成为挑战。现有系统存在两大痛点：
*   **架构缺陷**：要么基于 DataFrame 原语（如 Lotus），缺乏现代数据库（DBMS）的向量化和并行执行基础设施；要么将 LLM 调用封装为 SQL UDF（如 FlockMTL），导致语义逻辑对优化器透明，且增加了用户维护成本。
*   **优化难题**：LLM 调用成本极高且具有非确定性，传统的基于代价的优化模型（Cost Model）和重写规则在面对自然语言（NL）谓词时失效。
**Sema 的目标**是将语义算子视为数据库的“一等公民”，在 DuckDB 基础上构建一个端到端的高性能语义查询引擎。

### 2. 方法论：核心思想与关键技术
Sema 的核心思想是将语义算子深度集成到数据库的解析、优化和执行全生命周期中。
*   **SemaSQL 声明式方言**：允许用户在标准 SQL 中通过 `s'...'` 语法注入自然语言表达式（如 `WHERE s'{review} 是正面评价'`），实现语义与关系运算的无缝组合。
*   **逻辑层优化（LLM 辅助）**：
    *   **表达式压缩（EC）**：利用轻量级 LLM 移除 NL 谓词中的冗余信息，减少 Token 消耗。
    *   **谓词推导（PD）**：利用 LLM 识别 NL 谓词中的确定性逻辑，将其转化为标准 SQL 谓词（如将“有效评论”推导为 `review != 'nan'`）并下推，从而提前过滤数据，减少 LLM 调用。
*   **物理层执行优化**：
    *   **算子融合（Fusion）**：将连续的语义过滤或投影算子合并为一个 LLM 调用，减少数据往返。
    *   **提示词批处理（Batching）**：将多行数据组合进单个 Prompt，提高吞吐量。
*   **自适应查询执行（AQE）**：由于语义算子的选择率和延迟难以预估，Sema 在运行时通过三个阶段（表达式探索、路径探索、路径利用）动态寻找延迟与 Token 成本之间的**帕累托最优（Pareto-optimal）**执行路径。

*   **Benchmark**：设计了 20 个语义查询，分为单算子查询、连续语义过滤查询以及混合语义算子查询（涉及分类、摘要、关键词提取等任务）。
*   **对比方法**：

    *   **Palimpzest**：具有级联优化器的语义查询系统。
    *   **FlockMTL**：基于 DuckDB UDF 的实现方式。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了 20 组核心查询对比、针对 4 种优化配置的消融实验、逻辑优化（EC 和 PD）的有效性分析，以及 AQE 的案例研究。
*   **充分性评价**：实验设计较为充分。作者不仅对比了端到端的延迟，还深入分析了 Token 消耗、API 成本和结果质量（F1、准确率等）。通过本地和远程两种环境的对比，揭示了不同场景下优化策略的差异性，体现了客观性和公平性。

### 6. 主要结论与发现
*   **性能提升**：Sema 相比 Lotus 实现了 2-4 倍加速，相比 Palimpzest 实现了 2-10 倍加速，显著优于基于 UDF 的 FlockMTL。
*   **质量保持**：在大幅提升效率的同时，Sema 的输出质量（准确率/F1）与基准系统持平，甚至在某些任务中因算子融合带来的上下文复用而表现更好。
*   **环境敏感性**：实验发现，在远程 API 场景下，减少调用次数（融合/批处理）是优化核心；而在本地部署场景下，Token 长度对延迟的影响更敏感，AQE 能有效平衡这些权衡。

### 7. 优点（亮点）
*   **深度集成**：不同于外挂式的中间件，Sema 将语义算子嵌入 DuckDB 内核，充分利用了向量化执行和流水线并行。
*   **自适应能力**：引入 AQE 框架处理 LLM 的不确定性，能够根据用户偏好（延迟优先 vs 成本优先）动态调整策略。
*   **易用性**：SemaSQL 极大地简化了语义查询的编写，用户无需编写复杂的 UDF 代码。

### 8. 不足与局限
*   **算子覆盖有限**：目前主要针对一元（Unary）算子的融合，对于更复杂的跨流水线（Cross-pipeline）算子优化仍有待加强。
*   **提示词敏感性**：尽管有表达式压缩，但 LLM 对 Prompt 的微小变化依然敏感，可能导致结果在不同优化路径下出现波动。
*   **模型依赖**：逻辑层优化（谓词推导）本身依赖于另一个辅助 LLM 的调用，虽然次数少，但在极小规模数据集上可能引入额外开销。
