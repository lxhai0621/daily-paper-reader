## Abstract 

Despite the remarkable performance of large language models (LLMs) in text-to-SQL (SQL generation), correctly producing SQL queries remains challenging during initial generation. The SQL refinement task is subsequently introduced to correct syntactic and semantic errors in generated SQL queries. However, existing paradigms face two major limitations: (i) self-debugging becomes increasingly ineffective as modern LLMs rarely produce explicit ex-ecution errors that can trigger debugging signals; (ii) self-correction exhibits low detection precision due to the lack of explicit error mod-eling grounded in the question and schema, and suffers from severe hallucination that frequently corrupts correct SQLs. In this paper, we propose ErrorLLM , a framework that explicitly models text-to-SQL Error s within a dedicated LLM for text-to-SQL refinement. Specifically, we represent the user question and database schema as structural features, employ static detection to identify execution failures and surface mismatches, and extend ErrorLLM ’s semantic space with dedicated error tokens that capture categorized implicit semantic error types. Through a well-designed training strategy, we explicitly model these errors with structural representations, enabling the LLM to detect complex implicit errors by predicting dedicated error tokens. Guided by the detected errors, we perform error-guided refinement on the SQL structure by prompting LLMs. Extensive experiments demonstrate that ErrorLLM achieves the most significant improvements over backbone initial generation. Further analysis reveals that detection quality directly determines refinement effectiveness, and ErrorLLM addresses both sides by high detection F1 score while maintain refinement effectiveness.

## 1 Introduction 

Text-to-SQL generation aims to translate user questions (natural language questions) into executable SQL queries [ 17 ]. As the cross-ing point intersecting natural language understanding and data-base systems, text-to-SQL has been a long-standing research ques-tion [ 32 ]. Existing LLM-based text-to-SQL methods often struggle to understand complex questions, database schemas, and the precise structural syntax constraints required during initial SQL genera-tion [ 43 ]. In this case, text-to-SQL refinement becomes a critical and almost compulsory module that refines the generated SQL queries to improve accuracy [ 33 ]. Recent advanced text-to-SQL implemen-tations have carefully studied and designed specific modules to refine the initially generated SQL queries [ 47 ], which can be catego-rized into two mainstream paradigms. Self-debugging [8 ]: LLMs interactively refine their generated SQLs according to execution results that. Self-correction [ 37 ]: LLMs are triggered by elabo-rated prompt engineering and workflows to semantically analyze the generated SQLs. Figure 1 provides an intuitive example of both paradigms. Beyond integrating these modules into complete text-to-SQL workflows, the significance of the text-to-SQL refinement task has raised growing interest in the community towards designing specialized approaches for SQL refinement [1, 6, 40]. However, recent studies reveal significant limitations in current text-to-SQL refinement modules and methods [ 40 , 53 ]. As the most intuitive paradigm, self-debugging approaches refine SQLs using execution feedback as guidance. Nevertheless, with the rapid devel-opment of LLMs, the backbone SQL generation capability has been largely improved [ 33 ]. In our pilot experiments, the syntax errors that raise execution warnings occupy only 3% of all incorrect SQLs; the remaining SQLs do not raise explicit execution errors and thus cannot be refined by self-debugging paradigms. For self-correction, this paradigm relies on LLMs’ internal reasoning to identify and correct errors without external feedback. However, without explicit modeling of SQL errors grounded in the question and schema, LLMs

More critically, this low-precision detection leads to severe halluci-nation during the correction phase: when LLMs are prompted to correct a SQL that is already correct, they tend to comply with the correction instruction regardless, modifying the SQL and produc-ing an erroneous result [ 23 , 35 ]. We refer to this phenomenon as 

corruption , where originally correct SQLs are corrupted into incor-rect ones. As evidenced by our experiments, self-correction [ 36 , 45 ]achieves a commendable fixed rate but low overall improvement because the substantial number of false-positive detections triggers widespread corruption that offsets successful corrections. The limitations of both paradigms point to a common challenge: effective text-to-SQL refinement fundamentally requires accurate SQL error detection, yet neither self-debugging nor self-correction provides it [ 17 ]. Self-debugging relies on execution signals that cover only a small fraction of errors, while self-correction attempts detection and correction simultaneously in a black-box manner. We propose that a dedicated SQL error detection module that explic-itly identifies error types grounded in the question and schema is the key to enabling effective text-to-SQL refinement .In this paper, we propose ErrorLLM , a framework that ex-plicitly models text-to-SQL errors within dedicated error tokens for text-to-SQL refinement. Specifically, we first represent the user question and database schema as structural features, including a question-schema structure and abstract syntax trees for SQL. For SQL error detection, we employ a two-stage approach: static su-perficial detection first identifies execution failures and surface mismatches through deterministic rules; we then extend Error-LLM ’s semantic space with dedicated error tokens that represent categorized implicit semantic error types. Through a well-designed training strategy, we explicitly model these errors that enabling the LLM to effectively detect complex implicit errors by predict-ing dedicated error tokens. For text-to-SQL refinement, guided by the detected errors, we perform error localization and analysis to pinpoint error-relevant AST nodes and schema elements, and then conduct priority-ordered error-guided refinement by prompting LLMs. Extensive experiments on BIRD and Spider benchmarks demonstrate that ErrorLLM achieves the most significant improve-ments over backbone LLMs, and further analysis on the NL2SQL-Bugs benchmark verifies the effectiveness of fine-grained error type detection. Our contributions are summarized as follows: 

• We propose ErrorLLM , a framework that explicitly models text-to-SQL errors by extending the LLM’s semantic space with dedicated error tokens. Each token corresponds to a specific error type, enabling precise SQL error detection. 

• We design a comprehensive pipeline consisting of SQL er-ror detection and error-guided text-to-SQL refinement. The detection stage combines static superficial detection with LLM-based semantic detection; the refinement stage lever-ages error localization, analysis, and priority-ordered con-text to guide LLMs in producing targeted corrections. 

• Extensive experiments demonstrate that ErrorLLM achieves the most significant improvements over backbone LLMs on 

BIRD and Spider benchmarks. Further analysis verifies the effectiveness of fine-grained error detection and confirms that the quality of SQL error detection directly determines the effectiveness of text-to-SQL refinement.

## 2.1 Text-to-SQL Generation

𝑓 t2S : (𝑞, S) → 𝑦, (1) where 𝑦 is the ground-truth SQL query corresponding to the natural language question. Specifically, for LLM-based text-to-SQL [17]: 

ˆ𝑦 = LLM (I , 𝑞, S) , (2) where LLM denote a large language model; I is the set of text-to-SQL instructions; ˆ𝑦 is the predicted SQL query for 𝑞 .

## 2.3 Structural Representations in Text-to-SQL 

Recent studies indicate that LLMs struggle significantly to correct their own declarative SQL outputs [ 40 , 60 ]. We therefore employ structural representations, rather than flat textual input, to capture the complex relations involved in our framework. 

Question-schema Structure. We first build a structural represen-tation of the question and schema to support the following process. To bridge the gap between 𝑞 and S, we construct a question-schema structure (QSS), denoted as G = (V G, EG ). Concretely, G unifies two key structures: (i) the database structure ( 𝑅 S ) encoding in-trinsic relations in the schema (e.g., table/column relations, primary keys, foreign keys), and (ii) the linking structure ( 𝑅 Link ) aligning question phrases with relevant tables/columns [60]. 

VG = V𝑞 ∪ V S ; EG = ES ∪ E Link , (6) where V𝑞 denotes nodes corresponding to tokens/phrases in 𝑞 , and 

VS denotes schema nodes (tables and columns) in S.

(i) Database structure ( 𝑅 S ). We construct a schema structure 

GS = (V S, ES ) where ES contains structural relations within S,including primary- and foreign-key relations between columns [ 3 ]. 

(ii) Linking structure ( 𝑅 Link ). To connect G𝑞 and GS , we establish cross edges ELink that link question nodes to candidate schema nodes that map question phrases to their relevant tables or columns. The linking structure 𝑅 𝐿 can be constructed using existing graph-based algorithms [ 3, 4, 48 , 60 ], or obtained directly from a schema linking function 𝑓 SL : V𝑞 → 2VS from existing methods [ 37 , 57 ]. We detail the construction process in Appendix B.1. 

Abstract Syntax Tree for SQL. Instead of prompting LLMs to di-rectly refine the complete declarative SQL, we convert the predicted SQL into an abstract syntax tree (AST) 𝜏 ˆ𝑦 and conduct error-guided refinement on this structured representation. We provide an AST example in Figure 2, where each node of 𝜏 ˆ𝑦 corresponds to a syn-tactic unit in ˆ𝑦 , such as a SQL clause, operator or schema element. Formally, we define the AST as a tree: 

𝜏 ˆ𝑦 = (V 𝜏 ˆ𝑦 , E𝜏 ˆ𝑦 ), 𝑣 ∈ V 𝜏 ˆ𝑦 , (7) where 𝑣 is the node. To explicitly connect the structural representa-tion with the declarative SQL query ˆ𝑦 = ( ˆ𝑦 1, ˆ𝑦 2, . . . , ˆ𝑦 | ˆ𝑦 | ), we define an alignment function: 

𝜋 : V𝜏 ˆ𝑦 → 2{1,..., | ˆ𝑦 | } , 𝜋 (𝑣 ) ∈ { 1, . . . , | ˆ𝑦 |} , (8) where 𝜋 (𝑣 ) returns the token indices in ˆ𝑦 that correspond to the syntactic unit represented by node 𝑣 . This alignment ensures that error localization results in 𝜏 ˆ𝑦 can be deterministically mapped back to token spans in the declarative SQL query. 

## 3 Methods 

In this section, we present the proposed methods from a workflow perspective and detail the training process of ErrorLLM .

## 3.1 Modeling SQL Error in ErrorLLM 

To explicitly model SQL errors with structural representations, we map descriptive error representations into the LLM’s seman-tic space via dedicated tokens. With this modeling, the LLM can explicitly associate specific error types with incorrect SQLs given the structural context defined above. By learning these dedicated representations, the model decouples the error detection capability, while preserving its foundational text-to-SQL capabilities [27].

## 4 Experiments 

To comprehensively evaluate ErrorLLM , we design experiments around four research questions. RQ1 : How does ErrorLLM per-form in end-to-end text-to-SQL refinement compared with existing methods? RQ2 : How effective are the SQL error detection task and the text-to-SQL refinement task individually and jointly? RQ3 :How accurate is ErrorLLM ’s fine-grained error type prediction at the per-category level? RQ4 : What is the contribution of each key component to the overall performance? 

## 4.1 Experimental Setups 

Datasets. First, we conduct the text-to-SQL refinement experiments on two widely-used text-to-SQL benchmarks. (i) BIRD [26 ] is a challenging text-to-SQL benchmark, and we evaluate on the official development set (1,534 samples). (ii) Spider [ 56 ] contains standard databases for text-to-SQL evaluation, and we also evaluate on the official development set (1,034 samples). To further examine gener-alization capability, we additionally evaluate on three (iii) Spider variants : Spider-Realistic [ 10 ] (508 samples), Spider-Syn [12 ](1034 samples), and Spider-DK [ 13 ] (535 samples), which introduce perturbations in question phrasing, synonym substitution, and do-main knowledge requirements, respectively. Secondly, we conduct SQL error detection experiments on GPT-4o generated SQLs and a dedicated SQL error detection benchmark: (iv) NL2SQL-Bugs [33 ]contains expert-annotated instances that detail the SQL errors for semantically incorrect user questions. The experiments of NL2SQL-Bugs are also based on BIRD development set (2,018 samples). Since the pre-defined error types in NL2SQL-Bugs are largely different, we map our error taxonomy Λ to their categories for evaluation. Unless otherwise specified, RQ1 is evaluated on all three bench-marks (i)–(iii) across both backbones on (i), and across GPT-4o on (ii) and (iii); RQ2, RQ4, and the extended analyses in the Appendix are conducted on the BIRD development set; and RQ3 is evaluated on the dedicated eroor detection benchmark NL2SQL-Bugs .

Evaluation Metrics. We adopt the following metrics across the proposed research questions. (i) Execution accuracy (EX) [56 ]measures whether the predicted SQL returns the same result as the ground-truth SQL when executed against the database, serving as the primary metric for end-to-end text-to-SQL refinement evalua-tion (RQ1, RQ4). (ii) Detection accuracy (D-Accuracy) measures the correctness of binary error detection for each SQL sample. (iii) Detection F1 Score (D-F1) as the harmonic mean of Precision 

and Recall , evaluates the quality of SQL error detection (RQ2, RQ4). 

(iv) Fixed Rate (FR) measures the proportion of the truly incorrect SQLs that were detected (true positives, TP) that are successfully fixed after refinement (RQ2): FR = |{ ˆ𝑦 ∈ TP | Exec ( ˜𝑦, D) = Exec (𝑦, D)}| |TP | . (25) 

(v) Corruption Rate (CR) measures the proportion of falsely de-tected correct SQLs (false positives, FP) whose execution results change after refinement (RQ2): CR = |{ ˆ𝑦 ∈ FP | Exec ( ˜𝑦, D) ≠ Exec ( ˆ𝑦, D)}| |FP | . (26) 

(vi) Type-specific Accuracy (TSA) [33 ] measures per-category detection accuracy for fine-grained error type evaluation (RQ3).

𝐴 , (27) where |TP | is the number of truly incorrect SQLs detected (true positives), |FP | is the number of correctly predicted SQLs falsely flagged (false positives), and FR and CR are as defined in Section 4.1. This decomposition reveals a fundamental tension in text-to-SQL refinement: improving ΔEX requires simultaneously maximizing detection recall (to increase |TP |) and precision (to minimize |FP |), while maintaining a high FR and low CR. A critical insight from our experiments is that LLMs exhibit strong compliance with error-guided instructions during the text-to-SQL refinement task. When provided with a false error signal, LLMs tend to modify the SQL regardless of its original correctness, resulting in an empirically high CR. For example, Self-correction [ 37 ] achieves a commendable FR of 79.02% on its 205 detected errors, yet its overall ΔEX on BIRD 

is only ↑2.34%. This limited gain is attributable to the substantial false positives that lead to a large number of correct SQLs being corrupted, offsetting most of the gains from successful corrections. 

ErrorLLM addresses both sides of Equation 27 effectively. On the detection side, it achieves both the highest recall (76.22%) and competitive precision (80.12%); On the refinement side, it maintains a FR of 53.88% even at this large detection scale. This dual advantage in detection quality and refinement efficiency yields the highest 

ΔEX across all methods, demonstrating that explicitly modeling SQL errors is essential for effective text-to-SQL refinement.

## 5 Conclusion 

In this paper, we propose ErrorLLM , a framework that explicitly models text-to-SQL errors within a dedicated LLM for text-to-SQL refinement. We extend the LLM’s semantic space with dedicated error tokens, where each token corresponds to a categorized error type, and design a comprehensive SQL error detection approach that combines static detection with LLM-based semantic detection. Guided by the detected errors, we further design an error-guided text-to-SQL refinement pipeline with error localization, error anal-ysis, and priority-ordered correction to produce fixes. Extensive experiments on BIRD and Spider benchmarks demonstrate that Er-rorLLM achieves the most significant improvements over backbone initial generation, and is the only method that consistently improves both standard and strong backbones without corrupting originally correct SQLs. Evaluation on the NL2SQL-Bugs benchmark further shows that the fine-grained error detection of ErrorLLM achieves competitive type-specific accuracy against proprietary LLMs. Our analysis confirms that the quality of SQL error detection directly determines the effectiveness of text-to-SQL refinement, validating our core design of explicitly modeling SQL errors in the ErrorLLM 

as the foundation for effective text-to-SQL refinement. Zijin Hong et al.

## A Related Work 

LLM-based Text-to-SQL. LLMs have enabled two paradigms for text-to-SQL: in-context learning (ICL) and task-specific fine-tuning. ICL (prompt engineering) [ 41 ], has emerged as the dominant ap-proach for leveraging powerful proprietary and open-source LLMs, and early studies have surveyed both prompting strategies and underlying model architectures [ 31 ]. Techniques initially devel-oped for other NLP tasks, such as Chain-of-Thought (CoT) prompt-ing [ 49 , 50 ], have been successfully adapted to text-to-SQL and shown to substantially improve generation quality [ 28 , 44 ]. Build-ing on CoT, methods such as DIN-SQL [ 37 ], ACT-SQL [ 58 ], and DAIL-SQL [ 14 ] augment LLMs with richer exemplars or external knowledge sources [ 16 ], while other approaches employ multi-agent collaboration and iterative refinement [ 45 , 47 ] or dynamically construct demonstrations via online example generation [ 36 ]. Re-cent ICL-based systems, including SuperSQL [ 23 ], Dubo-SQL [ 46 ], TA-SQL [ 39 ], E-SQL [ 2], CHESS [ 45 ], and RSL-SQL [ 5 ], instantiate these ideas with distinct designs for schema interaction, demon-stration selection, and reasoning, collectively demonstrating the effectiveness of ICL for LLM-based text-to-SQL [17]. In parallel, fine-tuning [ 49 ] has emerged as a complementary line of work, particularly important for handling complex queries and domain-specific or real-world database environments. Recent ad-vances include CodeS [ 25 ], which proposes a two-stage framework for SQL generation and refinement; OmniSQL [ 24 ], which scales fine-tuning across diverse SQL generation scenarios; DTS-SQL [ 38 ], which decomposes schema understanding and SQL synthesis into dual models; XiYan-SQL [ 15 ], which integrates ensemble-based refinement; and SQL-R1 [ 34 ], which applies reinforcement learning to align generated queries with execution-level feedback. Together, advances in both ICL-style prompting and LLM fine-tuning are rapidly pushing the frontier of LLM-based text-to-SQL systems. 

Text-to-SQL Refinement. Despite the progress of LLM-based text-to-SQL, these systems still frequently misinterpret complex ques-tions and schemas, or violate syntactic constraints during first-pass generation [ 43 ], making a refinement stage an essential compo-nent in modern pipelines. Dedicated refinement frameworks such as MAC-SQL [ 47 ], SQLFixAgent [ 6], MAGIC [ 1], and SHARE [ 40 ]have been proposed to correct logical and semantic errors in gen-erated queries. Systematic analyses further reveal recurring error patterns [ 7, 33 ] and hallucination behavior [ 51 , 54 ] that motivate fine-grained evaluation benchmarks such as NL2SQL-Bugs [33]. Two paradigms dominate this landscape. Self-debugging [ 8] iter-atively revises SQL using execution feedback, but is fundamentally limited by the fact that only a small fraction of erroneous queries trigger explicit execution failures [ 33 , 53 ]. Self-correction [ 19 , 37 ]relies on the LLM’s internal reasoning to analyze and fix potential mistakes without external feedback. However, without explicit er-ror modeling grounded in the question and schema, these methods exhibit low detection precision [ 40 , 51 ] and may hallucinate cor-rections on already-correct SQL [ 23 , 40 ]. This over-eagerness to modify correct outputs (a phenomenon we term corruption) off-sets much of the gain from successful fixes, yielding limited net improvement despite seemingly high fix rates [19].

Question–schema Linking Structure. The linking stage con-structs the cross edges Elink between question entity nodes V𝑞 and schema nodes VS . With the recent advances in schema linking [ 5 ], we support two complementary strategies. Strategy 1) consumes an external schema linking result 𝑓 SL : V𝑞 → 2VS produced by existing text-to-SQL systems [ 37 , 57 ], which is the most widely adopted approach in modern pipelines. Since the text-to-SQL re-finement is naturally subsequent to the workflow with schema linking process already, using this strategy is practically reason-able and efficient. Strategy 2) derives the linking entirely from the RGAT-contextualized embeddings Q′ and S′ when no external linker is available. ## C.2 Training Data Setup 

We detail the construction of the ErrorLLM training data from the 

BIRD training set, following the synthesized pipeline in Section 3.1. 

Source and Preprocessing. The BIRD training set contains 9,428 question-SQL pairs. We run the backbone LLM (GPT-4o) on all training samples, achieving an execution accuracy of 52.38%, yield-ing 4,938 correctly predicted and 4,490 incorrectly predicted SQLs. The incorrect predictions serve as the primary source pool for er-roneous sample synthesis, while no-error samples are uniformly drawn from both ground-truth SQLs and correct predictions. 

Dataset Composition. The training set is constructed in two se-quential stages. First, LLM-assisted injection is conducted on the full pool of 4,490 incorrect predictions, producing 2,519 accepted annotated samples (acceptance rate: 56.1%). Second, rule-based per-turbation is applied to balance the error sample composition, con-tributing an additional 2,325 samples (1,099 single-error and 1,226 compound-error) to ensure structural coverage across all 12 error types. Finally, 4,653 no-error samples are uniformly drawn from the correctly predicted and ground-truth SQL pool (with unique ques-tion ID), producing a near-balanced 49%:51% correct-to-incorrect ratio in the final training set of 9,497 samples. Table 5 summarizes the full composition with detailed count and proportion. 

LLM-assisted Injection Details. All 4,488 incorrect predictions are submitted to GPT-3.5-Turbo ( AsiLLM ) for joint error annotation and SQL correction, following Equation 13. A sample is accepted only when its execution results are aligned with the ground-truth, ensuring that the annotation reflects a faithful understanding of the error semantics; the remaining 43.9% of candidates are discarded rather than relabeled to prioritize annotation quality. The LLM-assisted partition is conducted prior to rule-based perturbation, as it determines the target size for the rule-based complement. 

Rule-based Perturbation Details. Rule-based perturbation is cal-ibrated to complement the LLM-assisted partition, contributing 2,325 samples in total. ## D.1 Efficiency Analysis. 

Table 7 compares the token consumption of ErrorLLM against rep-resentative baselines across training and inference stages. During training, the proprietary LLM cost of ErrorLLM is a one-time over-head attributable solely to LLM-assisted data synthesis, not to model training itself. This cost is substantially lower than MAGIC [ 1], which relies on proprietary LLMs throughout its pipeline construc-tion. The most distinctive characteristic of ErrorLLM at inference is the remarkably low open-source LLM output token count (1.45 tokens per sample on average), a direct consequence of constrained decoding: the model is restricted to predicting only from the er-ror token vocabulary. The proprietary LLM cost of ErrorLLM is accordingly bounded by detection recall: only samples flagged as erroneous trigger LocLLM and RefLLM calls, keeping the average proprietary input tokens (966.4) competitive against methods that apply correction unconditionally. Overall, ErrorLLM achieves a favorable efficiency-accuracy trade-off: it attains the highest EX (as reported in Table 1) while consuming substantially fewer propri-etary tokens than most baselines at inference time. The token con-sumption calculation is implemented through the well-recognized titoken package 7.

## D.3 Further Ablation Study 

We provide a detailed per-variant analysis of the ablation study in Table 3, supplementing the summary in Section 4.5. Seman-tic detection is the core driver. Removing semantic detection causes the most dramatic degradation: D-F1 collapses from 78.12% to 12.21% ( ↓65.91), confirming that LLM-based error token pre-diction is the fundamental mechanism of ErrorLLM . Removing superficial detection has negligible impact (D-F1 ↓1.87, EX ↓0.26), indicating the semantic detector largely subsumes rule-based cov-erage. Detection degradation propagates to refinement. Each Detection-group component’s D-F1 drop positively correlates with EX decrease: removing constrained decoding reduces D-F1 by 4.40 and EX by 2.08%; removing semantic initialization reduces D-F1 by 1.91 and EX by 1.24%, consistently validating that detection quality directly determines refinement effectiveness. Structural components have amplified impact on refinement. Removing structural components (QSS, AST, or linking structure) causes mod-erate D-F1 reductions ( ↓4.59–5.68) but disproportionately large EX drops ( ↓2.61–3.52), as these representations serve a dual role in both error detection and downstream error localization, subtree extrac-tion, and schema subgraph extraction. LLM prediction behavior is essential for training data. Removing LLM-assisted injection causes a substantially larger drop (D-F1 ↓10.57, EX ↓4.10) than removing rule-based perturbation (D-F1 ↓1.77, EX ↓1.04), demon-strating that learning from realistic LLM prediction errors, rather than perturbing gold-standard SQL alone, is the dominant factor for generalization during inference based on LLM predictions.
